## Summary

Data scientists using the CodeFlare SDK need a simplified job submission workflow that eliminates the manual burden of creating, managing, and tearing down Ray Clusters. Today's multi-step process -- provision a cluster, submit a job, then remember to delete the cluster -- is error-prone and leads to orphaned clusters that waste significant compute resources. Users need a submit-and-forget experience where they define and submit a job, and the platform handles the full cluster lifecycle automatically.

## Problem Statement

Data scientists on RHOAI currently follow a three-step process to run distributed jobs through the CodeFlare SDK: (1) create a Ray Cluster, (2) submit a job to it, and (3) manually tear down the cluster after the job completes. This workflow creates two significant problems:

- **Wasted resources**: Users frequently forget to delete clusters after their jobs finish, leaving idle Ray Clusters consuming GPU and CPU resources indefinitely. In enterprise environments with shared cluster quotas, orphaned clusters block other teams from accessing compute.
- **Confusing user experience**: The separation between cluster management and job submission forces data scientists to understand infrastructure concepts that are irrelevant to their work. They want to focus on their training scripts and data, not on cluster lifecycle management. The Kubeflow Training Operator (KFTO) already provides a simpler job-centric paradigm for PyTorch jobs, making the CodeFlare SDK's cluster-centric approach feel inconsistent and outdated by comparison.

## Affected Customers

- Red Hat OpenShift AI (RHOAI) users running distributed training and batch inference workloads
- Enterprise data science teams using the CodeFlare stack for Ray-based distributed compute
- Partners leveraging distributed training capabilities on OpenShift as part of their AI/ML platform offerings

## Business Justification

This is a strategic investment to improve the CodeFlare SDK's competitiveness and reduce operational costs for RHOAI customers:

- **Cloud resource waste reduction**: Orphaned Ray Clusters represent real cost to customers running on cloud infrastructure. Automatic cleanup directly reduces their cloud spend and improves RHOAI's cost-efficiency story.
- **User experience alignment**: KFTO already offers a job-centric paradigm for training workloads. Bringing the same simplicity to the CodeFlare SDK creates a consistent, unified experience across RHOAI's distributed compute capabilities, strengthening the platform's positioning in enterprise AI/ML platform evaluations.
- **Competitive positioning**: Enterprise customers evaluating AI/ML platforms expect a streamlined job submission experience. The current multi-step cluster management workflow is a friction point in competitive evaluations against platforms that offer simpler abstractions.

## Acceptance Criteria

- [ ] Users can submit a distributed Ray job through the CodeFlare SDK without first creating a standalone Ray Cluster
- [ ] Ray Clusters provisioned for a job are automatically cleaned up after the job completes, fails, or is cancelled -- no manual teardown required
- [ ] Users can monitor job status and retrieve logs through the SDK using a straightforward job lifecycle (created, running, completed/failed)
- [ ] The job submission workflow integrates with existing Kueue-based workload queuing in RHOAI, so jobs respect cluster quotas and fair-sharing policies
- [ ] Users can submit jobs using default configurations without needing to build custom Ray container images for common use cases

## Success Criteria

- Data scientists can go from script to running distributed job in a single SDK call, matching the simplicity of KFTO's training job submission
- No orphaned Ray Clusters remain after jobs complete -- resource waste from forgotten clusters is eliminated
- The job lifecycle follows a simple, observable pattern (created, running, completed/failed) that users can monitor without understanding cluster internals
