## Summary

Platform engineers, SREs, and DevOps engineers need a command-line interface for managing Ray clusters and jobs through the SDK, so they can perform common operations (listing clusters, checking status, submitting jobs) without writing Python code and can integrate cluster management into CI/CD pipelines and shell-based workflows.

## Problem Statement

Today, all Ray cluster and job management operations require writing and executing Python scripts or using Jupyter notebooks. Even the simplest operations -- listing clusters, checking status, or submitting a job -- require users to author Python code. This creates several problems:

- **Persona lockout**: Platform engineers and SREs who work primarily in shell environments and are not Python-focused cannot effectively manage Ray clusters through the SDK. They are locked out of the tooling entirely.
- **CI/CD integration gap**: There is no way to integrate cluster management into CI/CD pipelines, shell scripts, or standard DevOps workflows. Automation requires wrapping Python scripts, adding unnecessary complexity.
- **Operational friction**: Quick operational checks (cluster status, job logs) that should take seconds instead require launching a Python environment, importing libraries, and writing boilerplate code.

Users currently work around this by writing ad-hoc Python wrapper scripts, but these are fragile, inconsistent across teams, and unsupported.

## Affected Customers

- **Pinnacle Investments** -- enterprise platform engineering team managing ML infrastructure
- **DataPulse** -- SRE teams operating Ray clusters at scale
- **Frontier Banking** -- DevOps engineers building CI/CD pipelines for ML workloads
- Enterprise platform engineering teams broadly, SRE teams managing ML infrastructure, and DevOps engineers integrating ML workloads into automation pipelines

## Business Justification

- **User base expansion**: Removing the Python-only barrier is expected to drive a 10x increase in non-data-scientist users (SREs, platform engineers, DevOps engineers) who can manage Ray clusters through the SDK.
- **Competitive table stakes**: Every major competitor provides CLI tooling -- AWS CLI, gcloud, az ml. Kubernetes ecosystem users expect CLI tools and will not adopt a Python-only SDK. This is not a differentiator; it is a requirement for enterprise adoption.
- **Automation simplification**: Expected 90% reduction in pipeline complexity for cluster management automation by replacing Python wrapper scripts with native CLI commands.

## Acceptance Criteria

- [ ] Users can list, inspect, create, delete, and scale Ray clusters from the command line without writing Python code
- [ ] Users can submit, list, inspect, stop, and retrieve logs for Ray jobs from the command line without writing Python code
- [ ] Command output is available in multiple formats (table for human readability, JSON and YAML for scripting and automation)
- [ ] Shell completion is available for bash, zsh, and fish to support discoverability and efficient use
- [ ] Users can create clusters from presets to simplify common configurations

## Success Criteria

- Quick status checks (cluster status, job logs) are at least 5x faster than the current Python script approach
- CI/CD pipelines can integrate cluster management using standard shell commands without Python dependencies
- All major cluster and job management operations are accessible from the command line
