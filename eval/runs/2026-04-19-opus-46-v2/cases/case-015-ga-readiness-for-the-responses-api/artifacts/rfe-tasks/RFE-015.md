## Summary

Red Hat AI customers cannot migrate production workloads from OpenAI to the RHOAI Responses API because the current Tech Preview implementation has incomplete field/parameter coverage, silently drops unsupported fields, and exhibits undocumented behavioral differences from the OpenAI Responses API specification. Full API parity is required so that customer applications written against the OpenAI Responses API work against Red Hat AI with zero code changes, enabling the TP-to-GA conversion that is the platform's primary revenue driver.

## Problem Statement

Customers building on OpenAI's Responses API -- the primary interface for agentic capabilities including tool use, multi-turn reasoning, and structured outputs -- cannot switch to Red Hat AI without significant rework. The current Tech Preview implementation:

- **Silently drops fields**: Unsupported request fields are accepted without error but ignored, causing unexpected behavior that is difficult to diagnose.
- **Missing parameters**: Key Responses API parameters are not yet implemented, forcing customers to remove functionality or maintain separate codepaths for OpenAI and Red Hat AI.
- **Undocumented behavioral differences**: SSE streaming event format, response structure, and error handling differ from OpenAI in ways that are not documented, causing runtime failures when customers point existing applications at Red Hat AI.

This forces customers into one of three unacceptable positions: maintain dual codepaths (one for OpenAI, one for Red Hat AI), accept unexpected production failures, or delay migration entirely. The entire "Private OpenAI" value proposition -- which is the core positioning for Red Hat AI -- depends on API parity.

## Affected Customers

- **Southern European Public Sector** (7 accounts: NARC, AGRI, NPSI, Posta Nazionale, PTI, ENER, GASI) -- positioning Red Hat AI as "Private OpenAI" for sovereign deployments; API parity is the entire value proposition for these accounts
- **Keystone Financial** -- requires OpenAI-compatible endpoints for their Model-as-a-Service deployment
- **Nexus Semiconductor** -- explicitly requires OpenAI API standard interface to enable provider portability ("easily switch models")
- **Meridian Bank** -- Backstage plugin integration depends on OpenAI endpoint parity
- **Aegean Mobile** -- pluggable RAG/agent stack requires standard Responses API support
- **Bosphorus Bank** -- requires OpenAI-compatible standard interface
- **VAS** -- actively migrating from Watsonx specifically because of OpenAI-style API limitations; incomplete parity in Red Hat AI would defeat the purpose of migration
- **15+ additional validated customers** across financial services, public sector, telecom, and manufacturing segments

## Business Justification

OpenAI API compatibility is the number-one adoption driver validated across customer discovery. This is not a speculative bet -- it is the single highest-leverage action to convert Tech Preview users into GA production customers.

- **Revenue at risk**: The 7 Southern European public sector accounts alone represent a major pipeline opportunity. Keystone Financial's MaaS deployment, Nexus Semiconductor's provider portability requirement, and VAS's Watsonx migration are each individually significant deals that are directly blocked by API parity gaps.
- **Strategic positioning**: Red Hat AI's competitive differentiation as "Private OpenAI" -- an OpenAI-compatible platform that runs in customers' own infrastructure -- is entirely dependent on actual API parity. Incomplete compatibility undermines the core value proposition.
- **Customer migration risk**: VAS is migrating from Watsonx specifically for better OpenAI compatibility. If Red Hat AI cannot deliver full parity, this customer and others like them have no reason to complete the migration.
- **Ecosystem effects**: As the Responses API becomes the standard interface for agentic AI capabilities, every tool, framework, and integration built against OpenAI's API becomes a potential Red Hat AI customer -- but only if compatibility is real, not aspirational.

## User Scenarios

1. **As an application developer at Nexus Semiconductor**, I need to point my existing OpenAI Responses API client (Python SDK) at Red Hat AI's endpoint and have it work without code changes, because my architecture depends on provider portability and I cannot maintain separate client libraries for each AI platform.

2. **As a platform engineer at Posta Nazionale**, I need Red Hat AI to provide the same Responses API behavior as OpenAI so that I can deploy a sovereign "Private OpenAI" service for my organization, because data residency requirements prevent me from using OpenAI directly and any behavioral differences would break internal applications built against the OpenAI standard.

3. **As an integration developer at Meridian Bank**, I need the Responses API streaming events from Red Hat AI to match OpenAI's SSE event format exactly, because my Backstage plugin processes the event stream and any format differences cause silent data loss or rendering failures in the developer portal.

## Acceptance Criteria

- [ ] A customer application written against the OpenAI Responses API (Python SDK) works against Red Hat AI's Responses API endpoint with zero code changes, or all differences are documented in a published compatibility matrix before GA
- [ ] Every unsupported Responses API field or parameter returns a clear, actionable error when submitted -- no silent field drops
- [ ] Streaming behavior (SSE event format, event types, event ordering) matches the OpenAI Responses API specification for all supported response types
- [ ] Conversations API integration works with the Responses API, enabling multi-turn interactions with the same semantics as OpenAI's implementation
- [ ] A published compatibility matrix documents per-field, per-parameter coverage compared to the OpenAI Responses API, with explicit status for each (supported, unsupported with error, planned)
- [ ] Performance benchmarks are published for representative Responses API workloads (latency p50/p95/p99, throughput, resource consumption) with sizing guidance for production deployments

## Success Criteria

The Responses API is considered GA-ready when:
- Customers can migrate existing OpenAI Responses API workloads to Red Hat AI without code changes, or with changes limited to what is documented in the compatibility matrix
- No production incidents are caused by silent field drops or undocumented behavioral differences
- The compatibility matrix is published and maintained as a living document that customers can reference before migration

## Scope

### In Scope
- Full OpenAI Responses API field and parameter parity for GA-supported capabilities
- SSE streaming format parity with OpenAI's Responses API
- Conversations API integration with the Responses API
- Compatibility matrix documentation (per-field, per-parameter coverage)
- Clear error responses for unsupported fields (replacing silent drops)
- Performance benchmarks and sizing guidance

### Out of Scope
- Production-grade response storage and persistence with retention policies and scale validation (recommended as a separate RFE)
- Multi-provider validation matrix across vLLM, remote::openai, remote::anthropic with automated CI testing (recommended as a separate RFE)
- Responses API observability integration with GenAI semantic conventions, MLflow/OTEL tracing, and dashboard templates (recommended as a separate RFE)
- TP-to-GA upgrade and migration path with documented migration steps, automated tooling, and QE validation (recommended as a separate RFE)
