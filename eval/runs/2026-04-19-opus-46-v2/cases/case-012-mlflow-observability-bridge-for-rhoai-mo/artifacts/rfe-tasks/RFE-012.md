## Summary

Data scientists and platform administrators running RHOAI need MLflow experiment metrics and MLflow server operational metrics to be available in the platform's monitoring pipeline. Today, training metrics (loss, accuracy, custom metrics) live exclusively in the MLflow UI while infrastructure metrics (GPU utilization, memory) live in the platform monitoring stack, forcing constant context-switching and making it impossible to correlate training performance with resource behavior. Administrators have no visibility into MLflow server health, storage consumption, or per-tenant usage from the centralized monitoring stack, leaving capacity planning and troubleshooting as guesswork.

## Problem Statement

Data scientists running experiments on shared multi-tenant RHOAI clusters cannot view MLflow training metrics alongside infrastructure metrics in a single correlated timeline. When a training run shows degraded loss convergence, they must manually cross-reference timestamps between the MLflow UI and infrastructure dashboards to determine whether the cause is a code issue or resource contention on the shared cluster. This context-switching is time-consuming and error-prone, particularly during time-sensitive POC evaluations.

Platform administrators responsible for shared MLflow infrastructure have no way to monitor MLflow server health, request latency, active experiment counts, or per-tenant storage consumption from the centralized monitoring stack. Without these operational metrics, administrators cannot proactively identify capacity constraints, troubleshoot degraded MLflow performance, or plan infrastructure scaling for growing tenant bases.

## Affected Customers

- **Sterling Bank Group** ML Platform team -- running shared multi-tenant RHOAI clusters
- **Kronos AI Factory** -- enterprise RHOAI deployment at scale
- **Central Telco** data science division -- multi-tenant cluster environment
- Enterprise RHOAI customers broadly: organizations running shared multi-tenant clusters where correlating training and infrastructure metrics is critical for production ML operations

Multiple large accounts have flagged this observability gap as a friction point during POC evaluations and expansion deal discussions.

## Business Justification

Unified monitoring is a table-stakes expectation for production ML platforms. The inability to correlate MLflow training metrics with infrastructure metrics represents a core platform observability gap that is actively blocking enterprise RHOAI adoption at scale. Multiple large accounts -- including Sterling Bank Group, Kronos AI Factory, and Central Telco -- have identified this gap as a friction point during POCs and expansion deals. Resolving this gap removes a barrier to enterprise adoption and strengthens RHOAI's positioning as a production-ready ML platform.

## Acceptance Criteria

- [ ] MLflow experiment and run training metrics (loss, accuracy, and custom metrics) are available in the platform monitoring pipeline, queryable by run ID and timestamp
- [ ] Training metrics from MLflow can be visualized alongside infrastructure metrics (GPU utilization, memory) in Grafana/Perses dashboards with time-correlated views per run
- [ ] MLflow server operational metrics -- including request latency, active experiment count, and per-tenant storage consumption -- are collected by the platform monitoring stack
- [ ] Platform administrators can monitor MLflow server health and per-tenant resource usage from the centralized monitoring stack without accessing MLflow directly
- [ ] Documentation covers how to set up and use MLflow metrics bridging into the monitoring pipeline

## Success Criteria

- Data scientists can diagnose whether poor training performance stems from code issues or resource contention without leaving the monitoring dashboard
- Platform administrators can proactively identify MLflow capacity constraints and plan infrastructure scaling using metrics from the centralized monitoring stack
- POC evaluations no longer cite "lack of unified monitoring" as a friction point

## Scope

### In Scope
- Bridging MLflow experiment and run training metrics into the RHOAI monitoring pipeline
- Exposing MLflow server operational metrics (request latency, active experiments, storage usage per tenant) to the monitoring stack

### Out of Scope
- OTEL collector pipeline deployment (separate RFE)
- Tenant-scoped metric query enforcement (separate RFE)
- Custom ML model drift/bias monitoring (separate RFE)
