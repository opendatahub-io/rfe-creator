## Summary

Users managing Ray clusters through the SDK need the ability to configure dynamic scaling behavior so that clusters automatically adjust their worker count based on workload demand. Today, clusters can only be created with a fixed number of workers, forcing users to choose between over-provisioning (wasting infrastructure spend during low-demand periods) and under-provisioning (causing performance bottlenecks and long job queues during peak loads). This capability should include scale-to-zero support so idle clusters stop consuming resources entirely.

## Problem Statement

When users create Ray clusters through the SDK, they must specify a static cluster size at creation time. This creates several pain points:

- **Over-provisioning waste**: Users who size clusters for peak demand pay for idle resources during low-demand periods. With 80% of Ray workloads exhibiting variable resource needs, the majority of clusters are running with significant waste.
- **Under-provisioning bottlenecks**: Users who size clusters conservatively experience long job queue times and performance degradation during peak loads, directly impacting productivity and time-to-results.
- **Manual intervention required**: Adjusting cluster size requires manual intervention -- users must monitor utilization, decide when to resize, and reconfigure clusters themselves. This operational burden is unsustainable at scale.
- **No idle cost elimination**: When clusters are not actively processing workloads, they continue consuming resources. There is no way to automatically scale a cluster down to zero workers and resume when new work arrives.

Current workarounds include manually monitoring clusters and recreating them at different sizes, or maintaining multiple clusters of different sizes for different demand levels -- both of which are error-prone and operationally expensive.

## Affected Customers

- **Quantum Analytics**: Runs large GPU training jobs with highly variable demand patterns; currently over-provisioning to handle peak loads
- **BioML Corp**: Runs batch inference workloads with spiky traffic patterns; experiencing long queue times during traffic bursts
- **CloudScale Inc.**: Operates a multi-tenant Ray platform and needs cost controls to manage shared infrastructure efficiently
- **Internal platform engineering teams**: Managing shared Ray infrastructure across the organization, spending significant operational effort on manual cluster sizing

Multiple enterprise prospects have cited the lack of autoscaling as a blocker for adopting the platform.

## Business Justification

- **Cost reduction**: Expected 60% reduction in infrastructure costs through scale-to-zero and right-sizing, directly impacting customer unit economics
- **Performance improvement**: Expected 90% reduction in job queue time during peak loads, improving user productivity and time-to-results
- **Operational efficiency**: Zero manual intervention for cluster sizing eliminates a significant operational burden for platform teams
- **Adoption blocker removal**: Multiple enterprise prospects have identified autoscaling as a prerequisite for adoption -- this capability directly unblocks new revenue
- **Market reach**: 80% of Ray workloads have variable resource needs, making this relevant to the vast majority of the user base

## User Scenarios

1. **As a data scientist at Quantum Analytics**, I need my Ray training cluster to automatically add GPU workers when I submit a large training job and release them when the job completes, because I don't want to pay for idle GPUs between training runs or wait in a queue because I sized my cluster too small.

2. **As a platform engineer at CloudScale Inc.**, I need to set cost controls on shared Ray clusters so they scale within defined boundaries (minimum and maximum workers), because I need to balance resource availability with budget constraints across multiple teams.

3. **As an ML engineer at BioML Corp**, I need my inference cluster to scale to zero when no requests are coming in and automatically spin up when traffic arrives, because our batch inference workloads are spiky and we're currently paying for idle clusters 70% of the time.

## Acceptance Criteria

- [ ] Users can configure `min_workers` and `max_workers` parameters when creating a Ray cluster through the SDK, including `min_workers=0` for scale-to-zero capability
- [ ] Users can provide an autoscaling configuration that defines scaling policies based on resource utilization (CPU, memory, GPU) thresholds
- [ ] Users can define scaling policies based on workload signals such as Ray task queue depth
- [ ] Users can define scaling policies based on custom metrics available via Prometheus
- [ ] Users can query the current autoscaling status of a running cluster to see the active scaling state
- [ ] Users can dynamically adjust autoscaling configuration on a running cluster without recreating it
- [ ] Clusters automatically scale up and down based on configured policies without manual intervention

## Success Criteria

- Users can create Ray clusters with autoscaling enabled and observe automatic worker count adjustments in response to changing workload demand
- Clusters configured with `min_workers=0` automatically scale to zero when idle and resume when new work is submitted
- Platform teams can set scaling boundaries that prevent runaway resource consumption while ensuring adequate capacity during peak loads
- No manual intervention is required for routine scaling operations after initial configuration

## Scope

### In Scope
- Autoscaling configuration for Ray clusters created through the SDK
- Resource-based scaling policies (CPU, memory, GPU utilization)
- Workload-based scaling policies (task queue depth)
- Custom metrics-based scaling policies (via Prometheus)
- Scale-to-zero and scale-from-zero capability
- Runtime status querying and dynamic configuration updates

### Out of Scope
- Autoscaling for non-Ray cluster types (Spark, standalone training jobs)
- Cluster federation or multi-cluster autoscaling coordination
- Cost reporting or chargeback integration
- Predictive/scheduled scaling (scaling based on time-of-day patterns rather than real-time signals)
