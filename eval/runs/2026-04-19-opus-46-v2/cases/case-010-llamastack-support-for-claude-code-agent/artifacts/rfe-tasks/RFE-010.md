## Summary

Customers building AI agents with the Claude Code Agent SDK need to deploy and orchestrate those agents through LlamaStack on OpenShift AI. LlamaStack currently lacks the API surface and state-handling capabilities required to support the Claude SDK's orchestration model, preventing customers from centralizing their agentic AI workloads on the platform. This compatibility gap blocks adoption of OpenShift AI for a critical and growing class of agentic workloads.

## Problem Statement

Customers using the Claude Code Agent SDK to build AI agents cannot deploy them through LlamaStack today. The SDK's orchestration logic requires three capabilities that LlamaStack does not currently provide:

- **Real-time token delivery:** The Claude SDK expects server-sent events (SSE) for streaming token responses during agent execution. Without this, agents cannot deliver real-time output to users, making interactive agentic workflows impractical.
- **Stateful session management:** Multi-turn agentic conversations involve sequences of tool calls where context must persist across turns. Today, the SDK must pass the full conversation history with every request because LlamaStack has no server-side session or thread management. This creates unnecessary overhead and breaks down at scale.
- **Tool definition interoperability:** The Claude SDK uses its own schema for defining tools (computer, text_editor, bash, and custom tools). LlamaStack's internal tool-calling structures use a different schema, and there is no translation layer between them. Customers cannot use their existing Claude SDK tool definitions with LlamaStack.

As a result, customers building agents with the Claude SDK are forced to either bypass LlamaStack entirely (losing unified audit trails, security filtering, and cost management) or abandon the Claude SDK in favor of less capable alternatives.

## Affected Customers

- **Atlas Telecom** -- top-10 OpenShift customer, actively building AI agents using the Claude Code Agent SDK and seeking to deploy them through LlamaStack on OpenShift AI.
- Broader segment of enterprise customers adopting agentic AI patterns who need to route agent traffic through a centralized inference gateway for governance, security, and cost control.

## Business Justification

Atlas Telecom's agentic AI workloads represent a significant revenue opportunity. Keeping Atlas Telecom on OpenShift and driving OpenShift AI adoption is a strategic priority. Centralizing agentic traffic through LlamaStack enables unified audit trails, security filtering, and substantial cost savings over third-party hosted models -- capabilities that Atlas Telecom requires for production deployment. Without Claude SDK compatibility, Atlas Telecom cannot consolidate their agentic workloads on the platform, putting both the account relationship and the broader agentic AI value proposition at risk.

## Acceptance Criteria

- [ ] Agents built with the Claude Code Agent SDK can be deployed and orchestrated through LlamaStack on OpenShift AI
- [ ] Real-time token delivery is supported via streaming, with time to first token under 200ms
- [ ] Multi-turn agentic conversations maintain context across 5+ turns without requiring the SDK to pass full conversation history on each request
- [ ] Claude SDK tool definitions (computer, text_editor, bash) are interoperable with LlamaStack's tool-calling interface without manual schema conversion
- [ ] Sessions can be resumed via session identifier after a network disconnect

## Success Criteria

- Atlas Telecom can deploy their Claude Code Agent SDK-based agents through LlamaStack in a supported configuration
- Context persistence verified via successful multi-turn conversations where LlamaStack retains state across turns without SDK-side overhead
- Streaming performance meets interactive latency requirements (time to first token under 200ms)
- Tool-call interoperability confirmed for the Claude SDK's standard tool definitions
