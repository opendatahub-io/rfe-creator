## Summary

Data scientists and platform teams need the ability to attach existing Kubernetes Secrets and ConfigMaps to RHOAI workbenches through the dashboard UI. Today, the workbench environment variables form only supports creating new inline secrets, which forces users with centralized secret management workflows (External Secrets Operator, HashiCorp Vault, GitOps/ArgoCD) to either duplicate secret values manually or bypass the dashboard entirely by editing Notebook CR YAML. This undermines enterprise secret management practices and creates adoption friction for organizations with mature security workflows.

## Problem Statement

The RHOAI dashboard workbench creation and editing UI only allows users to create new, inline secrets and config maps for environment variables. There is no option to reference secrets or config maps that already exist in the namespace.

This creates two problematic workarounds for enterprises that manage secrets centrally:

1. **Manual duplication**: Data scientists copy secret values from their centralized secret store and paste them into the dashboard's inline secret form. This breaks the single-source-of-truth principle, causes secrets to drift out of sync, and forces data scientists to handle raw secret values they should never need to see.

2. **Direct YAML editing**: Platform engineers edit the Notebook CR YAML to add `envFrom.secretRef` entries, bypassing the dashboard entirely. These externally-attached secrets are invisible in the dashboard UI, making workbench configuration opaque and unmanageable through the supported interface.

Both workarounds defeat the purpose of centralized secret management and reduce confidence in RHOAI as an enterprise-ready platform.

## Affected Customers

- **Skyline Aerospace** -- raised a formal support case for this capability
- **Multiple enterprise accounts** flagged by sales teams as blocked or friction-impacted during adoption
- **Organizations using External Secrets Operator with HashiCorp Vault backends** -- a common enterprise pattern where secrets are synced into Kubernetes Secrets from an external vault, and workloads consume them by reference

## Business Justification

Multiple active sales engagements have identified the inability to reference existing secrets as a blocker to RHOAI workbench adoption. Skyline Aerospace has escalated this through a formal support case. Enterprises with mature secret management infrastructure (External Secrets Operator, GitOps/ArgoCD, HashiCorp Vault) represent a significant segment of the RHOAI target market, and the current limitation creates friction in deals by undermining platform credibility for security-conscious organizations. Removing this gap directly supports enterprise adoption by aligning the dashboard with standard Kubernetes secret management practices.

## Acceptance Criteria

- [ ] When adding environment variables to a workbench, users can select existing Kubernetes Secrets and ConfigMaps from a namespace-scoped dropdown in the dashboard UI
- [ ] Secrets and ConfigMaps managed by external controllers (e.g., External Secrets Operator, ArgoCD) appear in the dropdown alongside manually-created ones
- [ ] The UI displays the keys within a selected Secret or ConfigMap (without exposing values) so users understand which environment variables will be injected
- [ ] Selected secrets are attached to the workbench via reference in the Notebook CR rather than by duplicating values
- [ ] When editing an existing workbench, the UI shows which existing Secrets and ConfigMaps are already attached
- [ ] Secrets and ConfigMaps created for one workbench can be reused in another workbench within the same project through the same selection mechanism

## Success Criteria

- Data scientists can configure workbenches with externally-managed secrets entirely through the dashboard UI, without handling raw secret values or editing YAML
- Platform teams can manage secrets centrally and have data scientists consume them by name, maintaining the single-source-of-truth principle
- The dashboard accurately reflects all secret attachments on a workbench, whether created inline or attached from existing resources
