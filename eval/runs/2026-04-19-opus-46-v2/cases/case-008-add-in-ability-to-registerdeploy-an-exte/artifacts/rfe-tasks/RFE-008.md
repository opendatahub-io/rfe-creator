## Summary

Platform administrators need the ability to register, view, edit, and delete external model endpoint registrations (such as OpenAI, Anthropic, Azure OpenAI, Bedrock, and Vertex AI) as MaaS-governed models directly from the RHOAI dashboard. Today, external model registration into MaaS is only possible through the CLI by creating MaaSModelRef CRDs, which creates an admin experience gap that undermines MaaS adoption and platform admin productivity. Adding dashboard UI for external MaaS model registration CRUD operations is required for MaaS GA completeness.

## Problem Statement

In RHOAI 3.4, external model endpoints can be registered into MaaS only through kubectl/CLI. Platform admins have no dashboard visibility into which external providers are configured, no ability to update or remove registrations from the UI, and no way to manage credentials at the MaaSModelRef level without CLI access. AI engineers consuming models through AAE cannot easily distinguish configured external MaaS models from other endpoint types.

This gap is fundamentally different from Custom Endpoints, which are unmanaged, ad-hoc connections without subscription binding or admin governance. MaaS external model registration requires admin ownership, MaaS subscription inclusion, and credential management -- all of which currently require CLI expertise. Enterprise customers expect full dashboard management for governed model endpoints, and CLI-only registration is a barrier to platform admin productivity, discoverability, and onboarding.

## Affected Customers

Enterprise platform admin teams adopting RHOAI MaaS for governed AI model access across large financial services, healthcare, and technology companies running OpenShift AI at scale. This includes major banks using Azure OpenAI through MaaS governance and Fortune 500 companies standardizing on RHOAI for multi-provider LLM access.

## Business Justification

This is a MaaS GA completeness requirement. The 3.4 release already supports external model registration via CLI and surfaces registered models in AAE, but the lack of dashboard UI creates an admin experience gap that undermines MaaS adoption at enterprise scale. Enterprise customers expect full dashboard management for governed model endpoints. CLI-only registration is a barrier to platform admin productivity, discoverability, and adoption -- particularly for organizations where platform admins may not have kubectl expertise or where security policies restrict direct cluster access.

## Acceptance Criteria

- [ ] Platform admin can create a new external MaaS model registration from the Model Deployments page, providing provider selection, endpoint URL, model name/identifier, and credential reference
- [ ] Platform admin can view all existing external MaaS model registrations from the dashboard, with clear identification of provider and registration status
- [ ] Platform admin can edit and delete existing external MaaS model registrations from the dashboard
- [ ] Registered external models appear in AAE for AI engineers with correct MaaS subscription access, consistent with CLI-registered models
- [ ] Existing CLI-based registration path continues to work -- dashboard UI is additive, not a replacement

## Success Criteria

Platform admins can perform the full lifecycle of external MaaS model registration (create, view, edit, delete) entirely from the RHOAI dashboard without requiring CLI access. AI engineers see registered external models in AAE with the same behavior whether registered via dashboard or CLI. UX is reviewed and approved by the design team before implementation.
