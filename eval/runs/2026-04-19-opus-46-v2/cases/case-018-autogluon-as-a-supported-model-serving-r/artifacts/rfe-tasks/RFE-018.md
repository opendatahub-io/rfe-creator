## Summary

Users who train models with AutoGluon -- an AutoML framework designed to make machine learning accessible to non-experts -- face a contradictory experience on OpenShift AI: training is simple, but deploying the resulting model requires hand-authoring complex ServingRuntime YAML and results in an explicitly unsupported custom runtime. AutoGluon should be available as a first-class, officially supported serving runtime in the OpenShift AI dashboard, on par with vLLM, OpenVINO, and Triton, so users can deploy AutoGluon models through the standard UI workflow without Kubernetes expertise.

## Problem Statement

Today, deploying an AutoGluon model on OpenShift AI requires users to manually obtain a KServe-compatible container image, navigate to Settings > Serving runtimes, and write or paste complex ServingRuntime YAML that demands knowledge of KServe specifications, API protocols, and resource configuration. The resulting custom runtime is explicitly marked as unsupported by Red Hat, meaning customers cannot get production support for it.

This creates a significant barrier for the primary AutoGluon audience -- citizen data scientists and data analysts who chose AutoML precisely because it eliminated complexity during model training. These users now face Kubernetes-level deployment complexity that undermines the entire value proposition of using an AutoML framework. Every other major serving framework in OpenShift AI (vLLM, OpenVINO, Triton) offers a streamlined select-and-deploy experience; AutoGluon is the notable gap.

Each manual deployment requires an estimated 2-4 hours of configuration work, and the unsupported status means enterprise customers with strict supportability requirements cannot adopt AutoGluon on the platform at all.

## Affected Customers

- **Enterprise data analysts and citizen data scientists** in financial services (fraud detection, credit scoring), healthcare (patient outcome prediction), retail (demand forecasting, churn prediction), and manufacturing (predictive maintenance) who use AutoGluon for tabular ML workloads
- **ISVs and partners** building AutoML-based solutions on OpenShift AI who need a supported runtime to offer their customers

## Business Justification

- The AutoML market is projected to reach $12.8B by 2030, and tabular data represents over 70% of enterprise ML use cases -- the primary domain where AutoGluon excels
- Native AutoGluon support attracts ease-of-use-focused users and increases overall platform adoption among personas who are currently underserved
- Eliminates 2-4 hours of manual configuration per model deployment, reducing time-to-value for every AutoGluon user
- Official Red Hat support removes the enterprise adoption barrier: customers who cannot use unsupported custom runtimes (due to compliance or organizational policy) are currently locked out entirely
- Parity with competing platforms that offer simpler AutoML model deployment experiences

## Acceptance Criteria

- [ ] AutoGluon appears as a selectable serving runtime in the OpenShift AI dashboard runtime dropdown, alongside vLLM, OpenVINO, and Triton
- [ ] Users can deploy an AutoGluon TabularPredictor model by selecting AutoGluon from the dropdown, specifying a model location, and clicking Deploy -- no YAML authoring required
- [ ] A Red Hat-verified container image for AutoGluon serving is published, maintained, and included in Red Hat support scope
- [ ] AutoGluon runtime is listed in the Red Hat OpenShift AI Supported Configurations document
- [ ] Official documentation is available, including a deployment guide, usage examples, API reference, and troubleshooting guidance

## Success Criteria

- AutoGluon model deployment follows the same streamlined workflow as vLLM, OpenVINO, and Triton -- no manual YAML configuration needed
- Enterprise customers who were previously blocked by the unsupported-runtime policy can deploy AutoGluon models with full Red Hat support
- Existing custom AutoGluon serving runtimes continue to function (backward compatibility maintained)
