## Summary

The Llama Stack project must be renamed before it reaches General Availability in RHOAI 3.5. Meta has withdrawn its investment, and Red Hat is now the primary driver of the codebase and community. Continuing to ship under Meta's "Llama" trademark creates confusion about project ownership and governance, and exposes Red Hat to long-term legal and marketing risk. A pre-GA rename is dramatically cheaper than a post-GA rename, which would require deprecation cycles, customer migration guides, breaking changes across integrations, and extensive documentation rewrites.

## Problem Statement

The current project name "Llama Stack" contains Meta's trademark, tying the project's identity to a company that is no longer a meaningful contributor. As Red Hat grows the community independently, this creates confusion about ownership and long-term governance. Once the project ships at GA, customers, documentation, integrations, Kubernetes CRDs, container images, and community tooling will all reference the current name. A future rename after GA would be a breaking change with broad blast radius -- every consumer of the project's API surface, container images, CRDs, CLI commands, and documentation would need to migrate.

## Affected Customers

- **RHOAI enterprise customers** adopting agentic AI workloads -- they will build integrations, automation, and operational tooling referencing the project name, all of which would break on a post-GA rename
- **Partners** integrating with the Responses API and OpenAI-compatible APIs -- partner documentation, SDKs, and tooling reference the project name
- **Upstream open-source community** contributors and users building on the project -- community identity, tutorials, blog posts, and third-party integrations all embed the name

## Business Justification

Removing dependency on Meta's Llama trademark protects Red Hat's independent brand identity and avoids long-term legal and marketing risk. Red Hat is now the primary driver of the codebase and community, so the project identity should reflect that ownership. The cost asymmetry is stark: a pre-GA rename is a coordinated one-time effort, while a post-GA rename would require deprecation cycles across the Kubernetes API surface, customer migration guides, backward-compatibility shims maintained indefinitely, partner re-certification, documentation rewrites, and community re-branding -- an order of magnitude more expensive in engineering time, customer disruption, and brand confusion.

## User Scenarios

1. **As an RHOAI platform administrator**, I need the project's Kubernetes CRDs, container image names, and operator namespace to reflect the actual project ownership, so that my team is not confused about whether this is a Meta product or a Red Hat product when they encounter it in cluster resources.

2. **As a partner integrating with the agentic AI stack**, I need a stable, community-owned project name before I invest in writing documentation and building tooling around the API, so that I don't have to rewrite everything if the name changes after GA.

3. **As an upstream community contributor**, I need the project identity to reflect its actual governance, so that potential contributors and adopters understand this is a community-driven project led by Red Hat, not a Meta side project.

## Acceptance Criteria

- [ ] A new project name is selected through an open community process and cleared by Red Hat legal for trademark availability
- [ ] All upstream repository names, package names, and Python import paths use the new name
- [ ] All downstream RHOAI operator CRDs, container image names in registry.redhat.io, Helm charts, and Quick Start templates use the new name
- [ ] All Kubernetes API group names, namespace defaults, MCP server tool names, CLI commands, and API endpoint paths use the new name
- [ ] All GA documentation references the new name as the primary identifier (the old name may appear only in migration or historical context notes)
- [ ] Backward-compatibility aliases for the old name are maintained for at least one release cycle
- [ ] A community blog post or announcement is published explaining the rename and providing a migration path for existing users

## Success Criteria

- No instance of "llama" or "llamastack" appears in the Kubernetes API surface, container image names, or customer-facing documentation at GA (except migration/historical context notes)
- All CI/CD pipelines and integration tests pass under the new name
- No customer-facing material ships at GA with the old name as the primary reference
- Old-name aliases function correctly for the duration of the backward-compatibility window

## Scope

### In Scope
- Community name selection process and legal trademark clearance
- Coordinated rename across upstream repos, packages, and import paths
- Downstream rename across RHOAI operator CRDs, container images, Helm charts, Quick Starts, Kubernetes namespaces, API groups, MCP tool names, CLI commands, and API endpoint paths
- All GA documentation updates
- CI/CD pipeline updates
- Backward-compatibility aliases for one release cycle
- Community announcement and migration guidance

### Out of Scope
- Rebranding of upstream projects or technologies not under Red Hat's control
- Changes to the project's technical architecture or functionality (this is a naming change only)
- Long-term maintenance of backward-compatibility aliases beyond one release cycle
